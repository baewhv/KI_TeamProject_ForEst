### 1. 브랜치 생성 규칙
- 브랜치 병합 후에는 반드시 삭제할 것
- 기능단위로 브랜치를 나눌지, 인원 단위로 나눌지 정할 것

### 2. 커밋 메시지 규칙
- [ADD] : 새로운 기능 생성 시
- [UPDATE] : 만들어 놓은 기능의 변경점이 있을 시
- [FIX] : 기능의 버그가 있어 수정했을 시
- [REFACTOR] : 코드 리팩토링 시
- ex) [ADD] Player 이동 로직 생성

### 3. Pull Request 거쳐서 병합할 것
- 병합하기 전 반드시 팀내에 공지할 것
- 팀내에서 3인 이상이 찬성할 시 병합할 것
- 가능한 팀장이 있을 때 병합할 것

### 4. Unity 프로젝트 폴더 구조
- 각 요소에 맞는 폴더 Scenes, Prefabs, Auido, Resources, Sprites 등등 미리 생성
- Imports 폴더를 만들어서 에셋 관리할 것

### 5. 네이밍 규칙(코드)
- 열거형 사용 시 앞에 대문자 E를 넣을 것
- 인터페이스 사용 시 앞에 대문자 I를 넣을 것
- 추상 클래스 사용시 앞에 Base를 넣을 것

### 6. Scene 관리방식
- 기획의도에 맞게 Scene을 여러개  생성한다면 Scene을 어떻게 관리할지 정할 것
- ex) SceneManager.Load - Scene 이름 or Scene 인덱스

### 7. 프리팹 네이밍

### 8. 작업 관리 도구
- 노션 제공
- 업무 상태 구분을 확실히 할 것 ( 예정 / 진행 중 / 완료 / 보류 )

### 9. 커뮤니케이션
- 디스코드
- 그외에 중요한 회의록이나 파일 등은 카카오톡으로 소통해도 좋음

### 10. 기획 - 개바 간 피드백 전달 방식
- 기획팀 빌드 테스트 후 피드백을 아마 깃허브 이슈에 전달
- 피드백을 즉시 반영할지, 검토후 수용 여부 결정할지 프로세스 정하기
