# 사전 협업 팀 프로젝트 그라운드 룰

- 기본 규칙
  - 급한 일이 생겼을 시 팀 내에 반드시 공지
  - 작업중인 스크립트 외에 연결이 있는 스크립트를 수정할 시 반드시 팀에 고지
- 회의
  - 매일 17시 30분 팀 전원이 모여 프로젝트 진행 상황을 공유
  - 기능 구현에 있어 막히는 부분이 발생하면 팀원이 모여 회의
- 목표
  - 4월 3일 첫 출시일까지는 기본 구현을 마치고 폴리싱, QA, 버그 픽스 작업을 진행할 수 있도록 예정

## Github 사용 규칙

### 커밋 메시지 규칙
- [ADD] : 새로운 기능 생성 시
- [UPDATE] : 만들어 놓은 기능의 변경점이 있을 시
- [FIX] : 기능의 버그가 있어 수정했을 시
- [REFACTOR] : 코드 리팩토링 시
- [DOCS] : 문서 수정 시
- [STYLE] : 코드 스타일 변경 시(코드 포맷팅, 코드의 기능 변경 없을 경우) 
- ex) [ADD] Player 이동 로직 생성

### 브랜치 생성 규칙
- 브랜치 병합 후에는 반드시 삭제
- 브랜치 네이밍은 '기능명_작업자 이니셜' 형식을 기본으로 설정

### Pull Request 병합 규칙
- 병합하기 전 반드시 팀 내에 공지
- 팀내에서 3인 이상이 찬성할 시 병합
- 가능한 팀장이 있을 때 병합

### Github Issues
- 버그 발생 시 Github Issues에 작성
- 작성 후 소통 창구를 통해 공지
- QA기간에 기획팀의 수정 사항 또한 Github Issues에 작성

## Unity 규칙
  
### 개발 담당 (미정)
- 김유훈
- 배정민
- 신하용
- 손지원
- 김덕환

### Unity 프로젝트 폴더 구조
- 각 요소에 맞는 폴더 Scenes, Prefabs, Auido, Resources, Sprites 등등 미리 생성
- Imports 폴더를 만들어서 에셋 관리할 것

### 네이밍 규칙(코드)
- 열거형 : E****            
  - ex) ECount
- 인터페이스 : I****        
  - ex) IAttackable
- 추상 클래스 : Base****    
  - ex) BaseMonster
- 멤버변수 : _camelCase
- 프로퍼티 : PascalCase
- 함수명 : PascalCase
- 매개변수 : camelCase

### Scene 관리방식
- 기획의도에 맞게 Scene을 여러개  생성한다면 Scene을 어떻게 관리할지 정할 것
- ex) SceneManager.Load - Scene 이름 or Scene 인덱스

### 정해진 프로그래밍 설계 방식
- 데이터의 관리는 구글 스프레드 시트로 하는 것으로 정함
- New Input System을 사용

## 작업 관리 툴 및 소통 창구

- 노션 제공

- 피드백을 즉시 반영할지, 검토후 수용 여부 결정할지 프로세스 정하기
